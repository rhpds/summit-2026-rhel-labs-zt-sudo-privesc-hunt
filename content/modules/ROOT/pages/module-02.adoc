= Finding the Misconfiguration

== Investigating Sudo Configuration

Now that we understand the situation, let's investigate the sudo configuration to find exactly what Manager Scott created.

=== Understanding Sudo

The `sudo` command allows permitted users to run commands as the superuser (or another user). The configuration is controlled by:

* `/etc/sudoers` - The main configuration file
* `/etc/sudoers.d/` - Directory for modular sudo rules (recommended for custom rules)

[TIP]
====
**Best Practice:**

Adding custom sudo rules to `/etc/sudoers.d/` is preferred over editing `/etc/sudoers` directly because:

* It keeps custom rules separate from system defaults
* It's easier to manage and review individual rule files
* It reduces the risk of syntax errors breaking the entire sudoers configuration
====

=== Checking What devuser Can Run

Let's start by checking what sudo privileges devuser has. We can do this using the `sudo -l` command, but we need to switch to the devuser account first.

First, let's see if we can switch to devuser:

[source,bash,role=execute]
----
sudo -i -u devuser
----

Now check what this user can run with sudo:

[source,bash,role=execute]
----
sudo -l
----

[NOTE]
====
The `sudo -l` command lists the commands a user is allowed (and forbidden) to run with sudo. This is extremely useful for security audits.

You should see output showing that devuser can run `/usr/bin/less` as root without a password.
====

Let's exit back to the rhel user:

[source,bash,role=execute]
----
exit
----

=== Finding Scott's Sudo Rule

Scott mentioned in his email that he created a rule in `/etc/sudoers.d/`. Let's look there:

[source,bash,role=execute]
----
sudo ls -la /etc/sudoers.d/
----

You should see a file related to devuser. Let's examine it:

[source,bash,role=execute]
----
sudo cat /etc/sudoers.d/devuser-logs
----

[IMPORTANT]
====
**The Vulnerable Configuration:**

```
devuser ALL=(root) NOPASSWD: /usr/bin/less
```

Let's break this down:

* `devuser` - The user this rule applies to
* `ALL` - Can run from any host
* `(root)` - Can run commands as the root user
* `NOPASSWD:` - No password required
* `/usr/bin/less` - The specific command allowed

**What Scott intended:** Let devuser view protected log files

**What this actually allows:** Full root shell access!
====

== Understanding the Vulnerability

The problem is that `less` is not just a simple file viewer. It's an interactive pager with many features, including the ability to execute shell commands.

=== Why This Is Dangerous

When you run a command with `sudo`, that command runs with root privileges. If that command can spawn a shell or execute other commands, those will ALSO run with root privileges.

The `less` command has several ways to execute commands:

* Press `!` followed by a command to execute it
* Press `v` to open the current file in an editor (which can also spawn shells)
* Use various other interactive features

[quote, Sysadmin Nate]
____
Yep, classic mistake. I've seen this pattern before.

Scott gave someone sudo to `find` once because "they just need to search for files." Didn't realize find has `-exec` that can run any command as root. That was a fun afternoon.

The problem is that a lot of Unix tools do way more than their name suggests. Viewers can edit. Editors can run shells. Search tools can execute commands. If you give someone sudo to ANY of these, you're basically giving them root access with extra steps.

There's actually a whole catalog of binaries that can be exploited like this - GTFOBins or something like that. Worth looking at before you approve any sudo request.
____

== Common Vulnerable Commands

Other common commands that can be exploited similarly:

* **Editors**: `vim`, `vi`, `nano`, `emacs` - Can all execute shells
* **Pagers**: `more`, `less` - Can execute commands
* **File operations**: `find`, `tar`, `zip` - Can execute commands via various flags
* **System tools**: `awk`, `sed`, `perl`, `python` - Are programming languages!
* **Others**: `man`, `git`, `ftp`, and many more

[IMPORTANT]
====
**Security Principle:**

When granting sudo access, you must understand the FULL capabilities of the command, not just its primary purpose. If a command can execute other commands or spawn a shell, it can be used for privilege escalation.
====

== Verification

Let's verify this is indeed the rule causing the problem. Check the file permissions and ownership:

[source,bash,role=execute]
----
sudo ls -l /etc/sudoers.d/devuser-logs
----

The permissions should be `0440` (read-only for root and the sudo group), which is correct for sudoers files.
